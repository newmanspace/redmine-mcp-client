# 代码重构 (Refactor)

本工作流是项目重构流水线的**主控引擎**。当你需要对现有代码进行重构、清理技术债务或优化架构时，触发此工作流。

## 目标与价值 (What it does)

它旨在将原本分散的重构立项、测试准备、分层执行、审查验证等环节，拼装成一个**自动升降档的智能状态机**。
通过这一个统一入口，系统会在后台自动执行测试先行、自下而上分层、架构一致性检查等红线防呆任务，最大限度降低重构风险。系统仅会在关键的、不可逆的决策点（如重构方案确认、合并前审查）将确认权交给你。

## 智能运作链路

触发该工作流后，系统将依次自主推进以下工作区块：

---

### 1. **立案与文档区块**

- **确认重构目标**：询问重构范围、问题和预期目标
- **读取参考文档**：扫描 `docs/development/refactoring/` 下的现有文档
- **生成重构文档骨架**：
  - 路径：`docs/development/refactoring/REFACTOR-XXX-<name>.md`
  - 参考：`REFACTOR-002-mcp-tools-layered.md` 模板
- **包含章节**：
  ```markdown
  ## 1. Background (背景)
  ## 2. Problem Analysis (问题分析)
  ## 3. Solution (解决方案)
  ## 4. Implementation Plan (实施计划)
  ## 5. Benefits (收益)
  ## 6. Testing (测试)
  ## 7. Migration Notes (迁移说明)
  ## 8. Success Metrics (成功指标)
  ```
- **[人工放行点]**：重构文档完成后，等待你的 Approve

---

### 2. **测试准备区块**

- **基线测试**：运行 `pytest` 确认当前测试全部通过
- **覆盖率检查**：识别目标模块的测试缺口
- **补充测试**：为即将重构的模块编写测试用例
- **[人工放行点]**：测试覆盖率达标后等待确认

---

### 3. **分层重构区块**

按 **自下而上** 顺序依次执行：

#### Step 1: Repository 层重构
- 调用 `refactor` 技能
- 重构数据访问层
- 运行单元测试验证

#### Step 2: Service 层重构
- 重构业务逻辑层
- 运行服务测试验证

#### Step 3: API 层重构
- 重构内部接口层
- 运行 API 测试验证

#### Step 4: MCP Tools 层重构
- 重构外部接口层
- 运行集成测试验证

**每层重构后**：
- ✅ 立即运行对应测试
- ❌ 测试失败则停止并报告

---

### 4. **双重内审区块**

- 调用 `refactor-review` 技能：
  - 审查重构文档完整性
  - 检查分层架构合规性
  - 验证测试覆盖率
  - 检查 git 提交顺序（是否自下而上）

- 调用 `code-review` 技能：
  - 代码白盒审查（SQL 注入、硬编码等）
  - 安全规范检查

- **[人工放行点]**：审查报告完成后，等待你的 Approve

---

### 5. **验证与终局**

- 调用 `test` 技能：运行全量测试和覆盖率检查
- 调用 `qa` 技能：审查重构文档排版和命名规范
- **更新文档**：
  - 标记 `REFACTOR-XXX-*.md` 状态为 `✅ Complete`
  - 更新 `docs/development/refactoring/README.md`
- **[人工放行点]**：汇总所有测试结果和审查报告，由你拍板是否调用 `git-commit` 技能进行规范提交

---

## 触发场景

| 场景 | 示例 |
|------|------|
| 代码坏味道 | 重复代码、过长函数、过大类 |
| 架构优化 | 分层不清、职责混乱、循环依赖 |
| 技术债务 | 临时方案、待重构 TODO |
| 性能优化 | 慢查询、N+1 问题 |
| 可测试性 | 难以单元测试、依赖难 mock |

---

## 重构原则（红线）

系统在执行过程中会严格坚守以下原则：

1. **测试先行** - 未写测试不动代码
2. **自下而上** - Repository→Service→API→MCP Tools
3. **文档先行** - 先立案后动刀
4. **覆盖率不降** - 重构后覆盖率 ≥ 重构前
5. **架构一致** - 无跨层调用、无循环依赖

## 与其他工作流的协作

| 工作流 | 关系 |
|--------|------|
| `feature` | 新功能开发完成后，可能需要 `refactor` 清理技术债务 |
| `issue` | Bug 修复后，可能需要 `refactor` 重构问题代码 |
| `/release` | 重构完成后，通过 release 流程发布上线 |

---

## 输出产物

重构完成后，将产出：

1. **重构文档**: `docs/development/refactoring/REFACTOR-XXX-<name>.md`
2. **测试用例**: 新增/更新的单元测试、集成测试
3. **审查报告**: Refactor Review Report
4. **更新摘要**: `docs/development/refactoring/README.md` 已同步
