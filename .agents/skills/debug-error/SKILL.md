# 错误调试 (Debug Error)

本技能专注于接管报错堆栈，系统性地分析错误信息与堆栈追踪，找出根本原因并提供修复建议。

## 前置约束
- 聚焦于实际报错的根因，而不仅仅是表面症状。
- 在深入代码前，优先排查最常见的低级错误。
- 提供的修复方案必须符合项目现有的架构与设计模式。
- **绝对禁止**在未经用户确认的情况下直接修改代码。

## 诊断流程

### 1. 收集错误上下文
向用户确认以下细节（如未提供全）：
- 完整的错误提示信息是什么？
- 有堆栈追踪（Stack Trace）吗？
- 是什么操作触发了这个报错？
- 这是一个新出现的 Bug 还是旧逻辑回归（Regression）？
- 报错能否稳定复现？

### 2. 分析报错特征
- **提取报错片段**：识别错误类型（TypeError, SyntaxError, NetworkError 等），定位具体报错的文件和行号。
- **推演堆栈路径**：梳理触发错误的调用链路，区分第三方库报错与内部业务层报错。

### 3. 排查常见问题字典
- `TypeError`: 访问了 null/undefined，或者传入了错误的类型。
- `ReferenceError`: 变量未定义或作用域隔离问题。
- `SyntaxError`: 拼写错误、括号缺失等语法不规范。
- `NetworkError`: 上游 API 宕机、CORS 跨域、认证失败或 URL 错误。
- `ModuleNotFound`: 依赖未安装或引用路径配错。

### 4. 深入代码诊断
- 自动读取报错指向的文件与行号周边代码。
- 如果是回归 Bug，优先检查最近的 Git 变更记录。
- 扫描上下游逻辑，寻找隐患。

### 5. 输出修复建议
按可能性高低输出解决方案（参见 `debug-error/examples/error-analysis-report-template.md`）：
- 详细解释哪里写错了。
- 摘录展示有问题的代码段。
- 提供修改后的**正确代码示例**。
- 解释为什么这样改能修好这个 Bug。

### 6. 验证修复
在应用修复方案后：
- 重新运行代码，检查报错是否解除。
- 观察是否引发了新的衍生错误。
